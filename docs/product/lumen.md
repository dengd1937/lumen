# Lumen

## 产品定义

**一句话描述**：面向咨询顾问的企业级 Agentic Deep Research 工作台，融合公司内部知识与公开网络数据，自主完成多源研究、交叉验证，产出带完整证据链的结构化报告。

**目标用户**：咨询顾问（行业研究 / 竞品分析 / 公司背调为主）

典型人设：MBB 某咨询顾问接到新能源汽车行业项目，客户要求 2 天内出初步洞察。

**产品形态**：Web 单页应用

技术栈：
- 前端：Next.js 15 + Tailwind v4 + shadcn/ui + React Flow（@xyflow/react）
- 后端：Python + LangGraph（fork `langchain-ai/open_deep_research`）+ FastAPI + SSE
- LLM：Qwen-Max / Qwen2.5-72B via DashScope API（模型层抽象成可切换，可 A/B 对比 Claude / GPT）
- 嵌入：DashScope `text-embedding-v3`
- 向量库：ChromaDB（本地）
- 抓取：Firecrawl API 或 LangGraph 内置
- 可观测性：LangSmith

**背景说明**：公司内部 AI 竞赛作品，1 人开发，1 个月交付 Demo。数据策略采用「方案 B」——用公开咨询报告（麦肯锡 / BCG / Gartner 白皮书）伪装成「内部知识库」+ 3-5 份针对 Demo 问题定制的虚构项目案例。

**核心场景**（5 分钟 Demo Script）：
1. 顾问输入：「研究中国新能源汽车行业 2024 年竞争格局及主要玩家」
2. 系统展示研究规划：拆成 5-7 个子问题（市场规模 / 主要玩家 / 政策 / 技术趋势 / 风险）
3. 实时展示**双轨道**检索过程：公开轨（Web 搜索 / 抓取）+ 私有轨（内部 KB 向量检索），并行进行
4. 3-5 分钟后产出结构化报告：摘要 + 分章节 + 数据表，每句话带引用角标
5. 报告中至少出现一处**「公开来源 vs 内部知识」冲突标注**（如"公开报告说市场规模 1.2 万亿，但内部 2023 年项目数据显示 8500 亿，口径不同"）
6. 点击任意引用角标 → 侧边栏展开原文片段高亮

## UI 范围

**涉及页面**：
- 研究输入页：问题输入框 + KB 状态指示 + 启动研究按钮
- 研究进行页：双轨道实时过程可视化（React Flow）+ 子任务状态列表
- 报告阅读页：结构化 Markdown 报告 + 引用角标 + 侧边栏原文浮窗 + KB 文档清单面板

**涉及组件**：
- 研究过程节点卡：展示单个子任务状态（规划中 / 检索中 / 完成）
- 双泳道时间线：公开轨 vs 私有轨并行进度，用不同色系区分
- 引用浮窗：点击角标弹出，展示原文片段 + 来源元数据
- 冲突标注块：内联嵌入报告正文，黄色高亮 + 双方来源对比
- 知识库只读面板：文档清单，研究进行时高亮被引用文档

**关键交互**：
- 问题提交 → 触发研究规划 → SSE 流式推送 → 双轨道节点实时渲染
- 点击引用角标 → 侧边栏滑入展示原文片段高亮
- 研究进行时 → KB 面板中对应文档节点高亮联动

## 用户流程

顾问在输入页填写研究问题（如「中国新能源汽车行业 2024 竞争格局」），点击「开始研究」。系统先展示任务规划（5-7 个子问题），随后进入研究进行页——左侧公开轨（Web 搜索/抓取进度）和右侧私有轨（内部 KB 向量检索进度）并行推进，每个子任务节点状态实时更新。约 3-5 分钟后，系统跳转报告阅读页，展示带完整引用角标的结构化报告（摘要 → 市场规模 → 主要玩家 → 技术趋势 → 风险），其中冲突信息用内联标注框突出显示。顾问可点击任意角标在侧边栏查看原始证据片段，也可在 KB 面板中查看哪些内部文档被引用。

## 功能清单

| 功能 | 优先级 | 简要描述 |
|------|--------|----------|
| F1 研究任务规划 | P0 | LLM 将问题拆解成可并行子任务树（固定 5-7 子任务模板 + LLM 少量定制） |
| F2 多源并行检索（混合） | P0 | Web 搜索 + 本地 KB 向量检索并行，合并 Top-K；混合检索是核心差异化底座 |
| F4 交叉验证与冲突标注 | P0（简化版） | 公开源 vs 内部知识冲突检测；MVP 用 Prompt + 精心设计 Demo 数据，不做通用实体对齐 |
| F5 结构化报告生成 | P0 | Markdown 长报告，固定 5 章节模板（摘要/市场/玩家/趋势/风险） |
| F6 引用追溯 | P0 | 角标 + 点击跳转侧边栏展示原文片段；信任感核心，评委 wow 时刻 |
| F7 研究过程可视化 | P0 | **双轨道**（公开 vs 私有）React Flow 实时进度图；与秘塔单轨拉开差异 |
| F8 私有知识库 Connector | P0 ⭐ | 启动时扫描本地目录 + ChromaDB 向量化，零管理 UI，抽象 Connector 接口；最高差异化 |
| F3 证据抽取精细化 | P1 | 从长网页抽相关片段并合并，MVP 粗抽取够用；V2 做语义去重、chunk 重排 |
| F9 知识库只读面板 | P1 | 展示 KB 文档清单，研究进行时高亮用到的文档；V2 做侧边栏高亮联动 |

**关于 7 个 P0 的说明**：这 7 个功能不是独立功能，而是一条完整 Demo 叙事链路上的 7 个不可替代节点，砍任何一个，5 分钟 Demo 故事讲不下去。解决「过多」的方式是严守每个功能的 MVP 简化边界。

**功能依赖关系**：

```
┌─ F8 私有 KB Connector ───┐
│                         ├─→ F2 混合检索 ─→ F3 证据抽取 ─→ F5 报告生成 ─→ F6 引用追溯
│  F1 任务规划 ────────────┘                    │
│                                              ↓
│                                          F4 冲突标注
│                                              ↓
└─→ F7 过程可视化（贯穿 F1-F5）────────────────┘

F9（P1）← 依赖 F8 完成后叠加
```

**明确不做的边界**：
- 知识库管理 UI（CRUD、权限、分类、版本）——运维层不做
- 真实公司数据接入（架构支持，Demo 不做）
- 多用户 / 账号体系 / 权限
- 报告编辑 / 协作 / 导出 PPT
- 历史研究管理 / 收藏
- 多轮对话 session（仅支持一次深度研究 + 最多一轮针对性追问）
- 金融 / 审计等其他行业定制

## 竞品参考

**竞品分类（3 大阵营）**：
- 通用消费级 DR：Perplexity Deep Research / OpenAI Deep Research / Gemini Deep Research
- 企业搜索平台：Glean / Hebbia
- 开源 DR 框架：gpt-researcher / LangChain open_deep_research

**核心启发**：
- Hebbia 的垂类叙事方法（针对金融 DD 场景定制）→ 对应 lumen 针对咨询场景定制
- 秘塔 AI 搜索的证据链 UI → 参考交互形式，但必须与其单轨拉开差异（升级为双轨）
- open_deep_research（LangGraph 官方）→ 本项目 fork 来源，delta 须清晰：F8 私有 KB + F4 升级版冲突标注 + F9 KB 面板

## 设计约束

- **LLM 约束**：默认使用 Qwen-Max / Qwen2.5-72B via DashScope API；模型层抽象为可切换，Demo 前可 A/B 对比
- **数据不出境**：ChromaDB 本地部署，向量数据不上传外部服务
- **Demo 数据策略**：「方案 B」——公开咨询报告伪装内部 KB + 3-5 份虚构项目案例文档；数据必须精心设计以触发冲突标注
- **单人交付**：1 个月，所有架构和 UI 决策必须对开发效率友好；避免过度工程化
- **React Flow 约束**：F7 双轨可视化通过 @xyflow/react 实现，需要设计明确的视觉语言（公开色 vs 私有色）
- **SSE 实时推送**：后端 FastAPI SSE → 前端渲染，前后端契约必须在设计阶段明确

## 竞品分析摘要

| 竞品 | 核心能力 | 我们的差异化 |
|------|----------|-------------|
| Perplexity Deep Research | 快速公开数据研究报告，引用清晰 | lumen 支持私有 KB 混合检索，输出企业专有视角 |
| OpenAI Deep Research | Agentic 规划 + 深度公开检索 | lumen 有冲突标注 + 过程可视化 + 私有数据融合 |
| Glean | 企业搜索 + 100+ SaaS 连接 | lumen 额外生成结构化研究报告（Glean 只做检索） |
| Hebbia | 金融/法律文档 DD 检索 | lumen 针对咨询垂类，有 Web 检索 + 报告生成 |
| 秘塔 AI 搜索 | 单轨研究路径图 + 证据链 | lumen 双轨（公开 vs 私有）+ 冲突标注（秘塔做不了） |
| gpt-researcher | 开源 DR，成熟度最高 | lumen fork open_deep_research，F8/F4/F9 为核心 delta |
| open_deep_research | LangGraph 官方多 agent 框架 | lumen 在此基础上增加私有 KB 混合 + 冲突标注 + 咨询垂类 |

**市场空白定位**：「公开 + 私有混合 DR + 咨询垂类场景」是明确空白——通用 DR 只做公开；企业搜索做私有但不生成研究报告；Hebbia 只做金融 DD 检索，不生成分析性研究报告。

## 技术约束与风险

- **开发者前端经验为零**：Next.js + React Flow 学习成本，用 Claude Code 辅助缓解；设计契约驱动前端开发以避免边想边写
- **LLM 质量不稳定**：Qwen 中文能力强但指令遵循不如 GPT-4，需 Prompt 精调；模型层抽象为可切换以便对比
- **Demo 数据依赖**：F4 冲突标注效果高度依赖精心设计的虚构项目案例文档，数据设计本身是工程
- **秘塔撞车风险**：F7 双轨可视化若视觉处理不当，评委可能认为与秘塔类似；设计阶段必须强化双轨差异化视觉语言
- **SSE 实时性**：LangGraph 节点执行时间不均，前端节点渲染与后端进度同步需要精细设计
- **fork delta 叙事**：必须在 Demo 中清晰说明 lumen 与 open_deep_research 的差异，否则会被质疑为「套壳」

## 成功指标

- **Demo 完成率**：5 分钟内完整走完「输入 → 双轨检索 → 报告 + 冲突标注 → 点击角标」全流程，无卡壳
- **冲突标注出现率**：Demo 问题（新能源汽车行业）必须触发 ≥ 1 处公开 vs 内部冲突标注
- **引用追溯响应**：点击角标后侧边栏展示原文高亮，延迟 < 500ms
- **评委理解度**：Demo 结束后评委能自行描述「公开轨 vs 私有轨」的差异（可视化达成效果）
- **竞赛目标**：Demo 进入竞赛决赛 / 获得评委认可

## 后续工作流

**路由决策**：Design Workflow L2（V2-1 → V2-4）→ Development Workflow

**理由**：lumen 涉及 3 个全新页面（研究输入页、研究进行页、报告阅读页）、全新交互模型（双轨道过程可视化、引用追溯面板、KB 高亮联动）、新 token 需求（双轨道视觉语言：公开色 vs 私有色；引用角标风格；冲突标注样式），且竞赛级 Demo 对 UI 质感有要求，不能走 L1 轻量化路径。

**设计阶段关注点**：
1. 双轨道视觉语言：公开轨 vs 私有轨的色彩系统、节点形状、连线样式
2. 冲突标注块：内联报告正文的视觉风格，需直观但不干扰阅读
3. 引用浮窗交互：角标点击 → 侧边栏动画 → 原文片段高亮的完整交互规范
4. SSE 推送 → React Flow 节点状态变更的前后端契约（设计契约驱动前端开发）

**消费本文档的 agent**：design-workflow skill V2-1（设计方向确定）
